MVCC: PostgreSQL Full-Row Copy vs MySQL/InnoDB Delta Storage
What happens when you UPDATE 1 column in a 5-column row
PostgreSQL (Full-Row Copy)
Original row:
id
name
email
city
age
UPDATE age = 31 ↓
New version (entire row copied):
id
name
email
city
age✓
Old version becomes dead tuple:
DEAD TUPLE — waiting for VACUUM
Impact: 5x write amplification per update. ~54% of updates also rewrite all secondary indexes. Dead tuples accumulate until VACUUM runs.
MySQL / InnoDB (Delta Storage)
Original row (clustered index):
id
name
email
city
age
UPDATE age = 31 ↓
Row updated in-place, undo log stores diff:
id
name
email
city
age✓
Undo log (compact delta):
undo: age=30 (only changed column)
Impact: Only changed column written + compact undo entry. Secondary indexes unchanged if PK untouched. Purge thread cleans undo logs automatically.
Based on: Andy Pavlo, "The Part of PostgreSQL We Hate the Most" (CMU, 2023)
